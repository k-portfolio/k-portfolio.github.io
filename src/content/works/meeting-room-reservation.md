## 概要

社内の会議室を Web から予約・確認・取消できる業務システムです。
「作ってみた」ではなく、**要件定義 → 設計（構成図・ER図・API・ADR）→ 実装 → 運用設計**までを一気通貫で行い、
システムエンジニアとしての上流工程・設計力・運用設計を示すことを目的に制作しました。
設計ドキュメントはリポジトリの `docs/` に一式まとめています。

## 課題

予約システムで最も重要な不変条件は「**同一会議室・同一時間帯のダブルブッキングを許さない**」こと。
ところが「既存予約を検索 → なければ登録」というアプリ層のチェックだけでは、
複数人が同時に予約した際、検索と登録の間に競合（race condition）が起きて重複が成立し得ます。
この整合性を、同時実行下でも確実に担保する設計が求められました。

## 工夫した点

- **競合制御を DB の排他制約で担保（設計の核）**: PostgreSQL の `EXCLUDE USING gist`（`tstzrange` ＋ `btree_gist`）で「同一会議室・時間帯の重なり」を物理的に禁止。アプリのバリデーションは突破され得るが DB 制約は最後の砦、という多層防御の考え方を採用し、**判断理由を ADR に記録**。
- **上流〜運用のドキュメント化**: 要件定義書／アーキテクチャ設計書／DB設計書（ER図）／API設計書／運用設計書（バックアップ・監視・障害対応）/ ADR を整備し、「なぜその設計か」を追える状態に。
- **3コンテナ構成**: nginx（リバースプロキシ）／FastAPI／PostgreSQL を docker-compose で統合し、app・db は外部に直接公開しない。
- **品質担保**: 業務ルールの単体テストに加え、**競合制御の DB 統合テスト**を用意。GitHub Actions で PostgreSQL を起動して CI 実行。
- **セキュリティ**: セッション＋CSRF、パスワードハッシュ、ロールによる最小権限、秘密情報の環境変数管理。

## 結果・学び

「要件定義で不変条件を言語化し、それを設計のどの層でどう守るか」を決めることが、SE の中核だと整理できました。
特に整合性を**アプリ任せにせず DB 制約に落とす**判断は、インフラ運用で培った「最後の砦を用意する」感覚と直結しています。
設計判断を ADR として残すことで、実装だけでなく「考えのプロセス」まで他者に説明できる成果物になりました。
